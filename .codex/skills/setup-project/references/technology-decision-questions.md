# Technology Decision Questions

`setup-project` 中に技術要素を対話で詰めるときの観点。

## 進め方

- 1 回で全部聞かない
- 2-4 個の論点ずつ確認する
- 迷っている場合は、推奨案と代替案を 1 つずつ出す
- 回答後は「決まったこと」「保留」を短く整理する

## 優先して聞く項目

1. フロントエンド
- Web / SPA / SSR / SSG のどれか
- React 継続か、別フレームワーク許容か
- UI ライブラリを使うか

2. バックエンド
- バックエンドは必要か
- 必要なら BFF / REST / GraphQL / RPC のどれか
- モノレポで持つか、別リポジトリか

3. データ
- 永続化は必要か
- 必要なら RDB / NoSQL / BaaS のどれか
- ローカル優先かクラウド前提か

4. 認証、認可
- ログインは必要か
- メール / OAuth / SSO のどれを想定するか
- 権限ロールを分けるか

5. 配布、運用
- どこにデプロイするか
- 開発環境、本番環境をどう分けるか
- 監視、ログ、分析は必要か

6. 制約
- 予算上限
- 運用できる人数
- 納期
- 既存資産や既存インフラ
- 避けたい技術、避けたいベンダー

## 対話の出し方

## 会話テンプレート

### 1. 技術対話の開始

最初の入り方の例:

```md
PRD をベースに、次は技術要素を決めます。
一度に全部は決めず、重要な論点から順番に詰めます。
まずは次の 3 点を確認します。

1. フロントエンドは SPA / SSR / SSG のどれを想定していますか
2. バックエンドは初期段階から必要ですか
3. データ保存は必要ですか。それとも MVP はフロントだけで閉じますか

迷っている項目はそのまま書いてください。こちらで推奨案も出します。
```

### 2. 推奨案を添える聞き方

ユーザーが未確定のときの例:

```md
現時点の要件なら、まずは React SPA + 必要最小限のバックエンドなし構成を推します。
理由は、初期の検証を早く回せて運用コストも低いからです。

ただし、次の条件があるなら別案に切り替えます。
- SEO が重要なら SSR
- サーバー側で秘匿処理が必要ならバックエンドあり
- 永続化や複数端末同期が必須なら DB あり

この前提で進めてよいですか。
```

### 3. 各ラウンド後の整理

回答を受けた直後のまとめ方:

```md
現時点の整理です。

- 決定: React SPA + Vite
- 決定: MVP ではバックエンドなし
- 保留: 将来の認証方式は OAuth を第一候補
- 保留理由: まずはコアフローの検証を優先するため

次は、認証、データ保存、デプロイ先の 3 点を詰めます。
```

### 4. 未決定が残る場合の進め方

```md
未決定の項目が残っていますが、後続文書を止めるほどではないので仮決定で進めます。

- 仮決定: 認証は MVP 外
- 仮決定: データ保存はローカルまたはモック
- 後で決める項目: 本番 DB、監視基盤

architecture と repository-structure では、この仮決定を前提として明記します。
```

### 5. 技術要素確定の確認

設計文書へ進む前の最終確認:

```md
技術要素の整理を反映すると、初期構成は次の想定です。

- フロントエンド: React + Vite の SPA
- バックエンド: 初期はなし
- データ: MVP では永続化なし、必要なら後で追加
- 認証: MVP 外
- デプロイ: 静的ホスティング前提

この前提で functional-design と architecture に落とし込みます。
問題なければ次へ進みます。
```

## 論点別の質問テンプレート

### フロントエンド

- 「利用形態は Web 前提ですか。それともネイティブやデスクトップも視野に入れますか」
- 「SEO や公開ランディングが重要ですか。重要なら SSR/SSG も候補に入れます」
- 「React は継続前提でよいですか。変える理由がなければ現状維持を推します」

### バックエンド

- 「サーバー側で保持したい秘密情報や重い処理はありますか」
- 「初期から API が必要ですか。不要ならフロント単体で始める案を推します」
- 「将来バックエンドを持つなら、REST と RPC のどちらがチームに合いそうですか」

### データ

- 「データは端末内だけで足りますか。それとも複数端末や共有が必要ですか」
- 「DB が必要なら、運用を軽くしたいですか。それとも柔軟性を優先しますか」
- 「MVP で扱うデータ量はどの程度を想定していますか」

### 認証、認可

- 「ログインが必要な利用者はいますか」
- 「外部アカウント連携が自然ですか。それともメールログインで十分ですか」
- 「権限ロールは一般ユーザーだけでよいですか。管理者ロールは必要ですか」

### 配布、運用

- 「公開先に制約はありますか。Vercel、Cloudflare Pages、社内基盤など候補はありますか」
- 「本番だけでなく、検証環境やステージングも必要ですか」
- 「最低限ほしい運用可視化は何ですか。ログ、監視、分析のどれが必要ですか」

### 制約

- 「月額コストの上限はありますか」
- 「誰が保守しますか。専任がいないなら運用の軽さを優先します」
- 「避けたい技術やベンダーがあれば先に固定したいです」

質問例:

- 「認証は必要ですか。必要なら、まずは OAuth 前提でよいですか」
- 「データ保存は必須ですか。それとも初期はローカル状態だけで進めますか」
- 「バックエンドを分ける理由がありますか。なければ初期はフロントエンド中心で閉じる案を推します」

回答後の整理例:

- 決定: React SPA + Vite を継続
- 決定: 初期はバックエンドなし
- 保留: 認証は MVP 外。将来 OAuth を候補にする
- 理由: まずは操作フロー検証を優先し、運用コストを増やさないため

# 🧪 テスト

テストの有効性は、統合テストと E2E テストによる包括的なカバレッジにあります。ユニットテストは個々のコンポーネントの検証に役立ちますが、アプリの信頼性は堅牢な統合テストと E2E テスト戦略によってこそ得られます。

## テストの種類

### ユニットテスト

ユニットテストは最も小さなテストです。アプリの個々の部分を単独で検証します。アプリ全体で使われる共有コンポーネントや関数のテスト、単一コンポーネント内の複雑なロジックのテストに有用です。実行が速く書きやすいのが特徴です。

[ユニットテストのコード例](../apps/react-vite/src/components/ui/dialog/confirmation-dialog/__tests__/confirmation-dialog.test.tsx)

### 統合テスト

統合テストはアプリの異なる部分が連携して正しく動作するかを確認します。テストの大部分は統合テストにすることで、アプリの信頼性に対する大きな自信が得られます。ユニットテストが通過しても、部品間の接続に問題があればアプリは正常に動作しません。機能をまたいだ統合テストによってアプリが一貫して正しく動作することを確保してください。

[統合テストのコード例](../apps/react-vite/src/app/routes/app/discussions/__tests__/discussion.test.tsx)

### E2E テスト

E2E（エンドツーエンド）テストはアプリケーション全体を評価する手法です。フロントエンドとバックエンドを含むアプリ全体を自動化し、システム全体が正しく機能することを確認します。実際のユーザーの操作をシミュレートします。

[E2E テストのコード例](../apps/react-vite/e2e/tests/smoke.spec.ts)

## 推奨ツール

#### [Vitest](https://vitest.dev)

Vitest は Jest に近い機能を持つ強力なテストフレームワークですが、より最新でモダンなツールとの相性が良いです。高いカスタマイズ性と柔軟性を持ち、JavaScript コードのテストで人気の選択肢です。

#### [Testing Library](https://testing-library.com/)

Testing Library はテストをかつてないほど簡単にするツール群です。実装の詳細ではなく、実際のユーザーと同じ方法でアプリをテストするという哲学を持っています。例えばコンポーネントの現在の状態値をテストするのではなく、ユーザーに対して何をレンダリングするかをテストします。状態管理ソリューションを変更するリファクタリングを行っても、コンポーネントの出力が変わらなければテストは有効のままです。

#### [Playwright](https://playwright.dev)

Playwright は E2E テストを自動化するツールです。実際のユーザーが実行するすべての操作を定義してテストを実行します。2つのモードで起動できます:

- **ブラウザモード** — 専用ブラウザを開いてアプリを最初から最後まで実行。各ステップでアプリを可視化・検査できるツールセット付き。開発時のローカル実行に向いています。
- **ヘッドレスモード** — ヘッドレスブラウザでアプリを実行。デプロイごとに実行する CI/CD への統合に便利です。

#### [MSW](https://mswjs.io)

API のプロトタイピングには MSW を使用してください。サーバーを気にせずフロントエンドを素早く構築できる優れたツールです。実際のバックエンドではなく、Service Worker 内のモックサーバーとして全 HTTP リクエストをインターセプトし、定義したハンドラーに基づいてレスポンスを返します。バックエンドの機能が未実装でもフロントエンドの開発をブロックされることなく進められます。

ハンドラーにビジネスロジックを実装することで API エンドポイントの設計にも活用できます。

[API ハンドラーのコード例](../apps/react-vite/src/testing/mocks/handlers/auth.ts)

[データモデルのコード例](../apps/react-vite/src/testing/mocks/db.ts)

完全に機能するモック API サーバーはテスト時にも便利です。fetch をモックする必要がなく、アプリが期待するデータをモックサーバーにリクエストできます。

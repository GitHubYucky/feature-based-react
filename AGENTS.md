# AGENTS.md

## 目的

このファイルは、このリポジトリで作業するコーディングエージェント向けの共通ルールを定義するものです。
ユーザーが明示的に別指示を出さない限り、このルールを全依頼に適用します。

## プロジェクトの優先事項

- コードと docs の整合性を保つ。
- 広いリファクタより、小さく目的が明確な変更を優先する。
- 明示的な要求がない限り、現在の feature-based 構成を維持する。
- 正しさ、保守性、分かりやすさを改善する実務的な変更を優先する。

## 作業方針

- 編集前に関連ファイルを読む。
- 新しい書き方を持ち込む前に、周辺の既存パターンを理解する。
- 可能なら新しい並行構造を作らず、既存構造を拡張する。
- リポジトリ内に現在のアプリと一致しないテンプレート由来の資料がある場合、明示指示がない限り現行実装を優先して判断する。

## 変更範囲

- 依頼を満たすために必要な最小限の変更にとどめる。
- 直接の妨げになっていない無関係な cleanup は行わない。
- 明確な理由がない限り、ファイルの rename、move、delete は行わない。
- より大きいリファクタの方が望ましい場合でも、必要がなければ実施せず、提案にとどめる。

## 安全ルール

- ユーザーの変更は、明示的に依頼されない限り巻き戻さない。
- `git reset --hard` のような破壊的な Git 操作や、同等の巻き戻し操作は、明示依頼なしに行わない。
- 未使用に見えても、意図的に残されている可能性があるため、確認なしに削除しない。
- `.claude`、`docs`、ルートの設定ファイルはノイズではなく、意味のある管理対象として扱う。

## コードルール

- 既存スタックである Vite、React、TypeScript、React Router、Tailwind に従う。
- import と命名は周辺コードに合わせて一貫させる。
- `console.log` のようなデバッグコードは残さない。
- 未完成の仮コードを完成済みとして置かない。
- 共通 utility を追加する場合は、現在のリポジトリ構成に合う場所へ配置する。

## ドキュメントルール

- 実装変更が挙動に影響する場合、必要に応じて関連 docs も更新する。
- docs と現行コードが矛盾している場合は、その事実を明確に示す。
- 次の 3 つを区別して扱う。
- 現在の実装
- 目標アーキテクチャ
- テンプレートまたは過去のガイド

## 検証

- 意味のあるコード変更後は、可能な範囲で関連する検証を実行する。
- まずは次のような最小限で有効な検証を優先する。
- `npm run build`
- `npm run lint`
- 存在するなら対象テスト
- 検証できなかった場合は、その理由を明記する。
- 既存不具合で検証が失敗した場合は、その点を明確に報告する。

## 出力方針

最終回答は簡潔にし、少なくとも次を含める。

- 何を変更したか
- 何を検証したか
- 残っている問題や制約

必要に応じて、次も示す。

- 置いた前提
- 作業中に見つかった既存問題

## 判断ルール

- 依頼が曖昧な場合は、最もリスクの低い解釈を優先する。
- 影響範囲が広い判断になる場合のみ、必要に応じて確認する。
- 安全に直接修正できる問題は、説明だけで終えず、可能なら修正まで行う。

## このリポジトリ固有の注意点

- このリポジトリには、次の 3 つの意味のある層がある。
- `src` のアプリケーションコード
- `docs` のドキュメント
- `.claude` のエージェント運用資産
- 変更時は、これらの層の整合性を意識する。
- `docs` が常に `src` の実態を正しく表しているとは限らない。
- `.claude` のワークフローファイルは使い捨てではなく、リポジトリ設計の一部として扱う。
